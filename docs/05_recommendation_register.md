# 05 - Recommendation Register

## Interpretation

The actions below are analyst proposals for a similar regulated bank environment. They are not statements about BSI's internal plan and are not BSI commitments.

## Priority definition

- `P1`: directly supports incident closure, regulatory evidence, recovery assurance, or personal-data impact decisions.
- `P2`: strengthens repeatability, cross-party coordination, communication consistency, or management oversight.

## Proposed sequence

| ID | Priority | Horizon | Action | Closure evidence |
|---|---|---|---|---|
| R-001 | P1 | 0-30 days | Publish a sanitized post-incident report. | Approved report and claim register. |
| R-002 | P1 | 0-30 days | Complete a forensic closure pack. | Evidence index, closure memo, independent validation. |
| R-003 | P1 | 0-30 days | Measure channel-specific recovery performance. | RTO/RPO and actual recovery scorecard. |
| R-005 | P1 | 0-30 days | Complete personal-data impact assessment. | Impact assessment and notification decision evidence. |
| R-004 | P1 | 31-90 days | Run a ransomware scenario exercise. | Test results, lessons learned, tracked actions. |
| R-006 | P2 | 31-90 days | Standardize incident status and communications. | Taxonomy, approval workflow, message matrix. |
| R-007 | P2 | 31-90 days | Map and jointly test critical third-party dependencies. | Dependency map, contract checklist, joint test result. |
| R-008 | P2 | 31-90 days | Create board-level lessons-learned metrics. | Monthly dashboard with owners and overdue actions. |

## Validation principle

A recommendation is not closed because a policy was written. Closure needs evidence that the action was implemented, tested, reviewed, and accepted by an accountable owner.

The complete register, evidence basis, reference IDs, and suggested ownership model are in [`../data/recommendation_register.csv`](../data/recommendation_register.csv).
